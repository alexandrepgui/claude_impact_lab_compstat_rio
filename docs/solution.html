Resumo executivo

"Antes de fazermos os plus a gente tem que fazer bem o feijão com arroz. O objetivo principal é o BINGO. O BINGO é o cruzamento dos dados de forma bem feita e confiável; a intenção das áreas não é estática, é dinâmica e se move ao longo do tempo."

O que estamos construindo: pipeline ETL — foco em Transform com human-in-the-loop — que cruza 5 fontes do CompStat, detecta inconsistências e produz Relatórios Analíticos em .doc por área FM. Substitui o relatório que a Duda monta à mão pra reunião de terça.

Lógica do score — motor BINGO semanal (configurável)

Update 2026-05-24 ~13:00: motor evoluiu de "média uniforme × n_camadas/3" pra soma ponderada por camada × categoria com janela temporal móvel. Config em shapefiles_qgis/config_pesos.json, código em shapefiles_qgis/motor_bingo_semanal.py. Tudo parametrizável (não tem fórmula hardcoded no script).

1. Estrutura do score

Por célula × semana, o score é:

bingo[célula, semana] = Σ (peso_camada × Σ (peso_categoria × n_eventos_normalizados))
                                     ↑                ↑
                            ocorrencias=1.0     dentro de cada camada,
                            disque=0.6          categorias têm pesos próprios
                            fatores=0.4         (default 1.0; alguns boost)

Score normalizado por percentil 95 dentro da janela pra reduzir efeito de outliers. Resultado: célula com sinal forte em 1 camada não atropela; valor alto exige convergência ponderada.

2. Parâmetros configuráveis (config_pesos.json)

ParâmetroValor atualO que controla

min_share_zona0.01Share mínimo (1%) pra zona entrar na alocação — descarta ruído

5. Janela temporal móvel (semanal)

Score recalculado a cada semana usando a janela das 8 semanas anteriores. Vantagens:

  • Áreas dinâmicas: quando crime migra (FM aciona um perímetro, crime vaza pra adjacente), o score atualiza naturalmente — sem hardcode
  • Sazonalidade: mês ruim em um ano não trava o score por anos
  • Animação no QGIS: visualizacao_semanal.html mostra evolução temporal — judges + Prefeitura veem o "filme", não foto

6. Alocação dos 600 agentes (output operacional)

Por semana, o motor escolhe as zonas ótimas de atuação da FM e distribui os 600 agentes proporcional ao score. Output em shapefiles_qgis/distribuicao_fm/zonas_semanais.shp. Cada feição = uma zona daquela semana com geometria recalculada, score agregado, nº de agentes alocados e semana de referência.

7. Como o LLM (DD desestruturado) se pluga aqui

O motor lê DD pelo campo tipo_pr. Hoje esse campo já vem do CSV plano (com só 2 classes top). Quando step 3 (LLM extraction) terminar:

  • LLM classifica cada relato em um id_canonico do vocab
  • Mapeia id_canonico → tipo_pr consistente (ou substitui tipo_pr pelo id_canonico direto)
  • Motor não muda — só lê dado mais rico
  • Pesos por categoria ganham granularidade (em vez de só "CONSUMO DE DROGAS", separa "consumo em cena pública crônica" vs "consumo eventual")

8. Extensibilidade — adicionar nova camada

O motor é plug-in (sem refactor). Para adicionar uma camada nova (ex.: 1746 BigQuery, Twitter):

  1. Gera shapefile de pontos da camada em shapefiles_qgis/<nova_camada>/
  2. Acrescenta entrada em LAYER_SPECS do motor (path, campo de data, campo de categoria, se temporal)
  3. Acrescenta bloco em config_pesos.json com peso da camada + pesos por categoria
  4. Roda python motor_bingo_semanal.py

Padrão consistente com o princípio "feijão com arroz primeiro": novas fontes (plus) entram sem mexer no motor.

Pipeline Transform — design detalhado

Princípio: pipeline 100% automatizada mas auditável por humanos. Cada passo deixa rastro. Inconsistências viram fila de revisão (auto-resolvida por IA quando seguro · escalada pra humano quando ambígua). Resultado: base consolidada confiável sobre a qual o score do BINGO é calculado.

1. Fluxo da pipeline (visão de alto nível)

P1

Load + Normalize (programático)

Ler cada fonte com seus encodings/separadores corretos. Padronizar CRS (WGS84), corrigir eixos invertidos (fatores_urbanos), renomear colunas pro schema canônico.

P2

Detect inconsistências (rule engine + LLM)

Rodar bateria de regras (geo bbox, completude, cross-source). LLM judga casos ambíguos. Output: lista de inconsistências com severidade + sugestão de correção.

P3

Resolve (auto-fix OU fila humana)

Se confiança alta + correção segura → auto-fix com log. Se ambíguo ou destrutivo → fila pra humano decidir. Pipeline pausa pra essa área até resolução.

P4

Enrich (programático + LLM)

Geo-intersect tudo com os polígonos FM. LLM extrai padrões de DD relatos + RELINTs (modus operandi, rotas fuga, recepção, horários, controle territorial). Filtra fatores urbanos pelos que importam pra dinâmica.

P5

Consolidate (programático)

Construir "área-snapshot" — registro único por (área FM × período) com todas as camadas integradas. Esse é o input pro score do BINGO.

P6

Audit trail (sempre)

Cada passo escreve: inconsistências detectadas, decisões tomadas (auto vs humano), confidence scores, timestamps. Humano pode reabrir e re-rodar.

2. Linguagem unificada — schema tabular (star schema) + vocabulário canônico

Update 2026-05-24 ~12:30: integrado o trabalho de análise qualitativa do colega (ver docs/analise_qualitativa_ocorrencias.md). Vocabulário canônico de 3 perspectivas + ~60 IDs canônicos passa a ser parte do schema. Os 7 pontos de sobreposição FU↔DD mapeados (OV-01 a OV-07) são os hotspots do BINGO por excelência — onde condição ambiental e evento criminal coincidem geograficamente.

A base consolidada é uma fact table flat com PK composta (area_fm_id, grid_cell, periodo). Cada linha = uma célula de uma área em um período. Colunas são contagens por tipo de ocorrência + métricas derivadas. Qualitativo (dinâmica de LLM, RELINTs) e referência (vocab canônico + matriz fator×órgão) entram como tabelas auxiliares joinadas por essas chaves.

Vocabulário canônico (3 perspectivas)

Toda entrada do vocab carrega um tipo_perspectiva que define a lógica de classificação:

FA FATOR_AMBIENTAL

Condição física do espaço que facilita crime. Lógica CPTED (prevenção pelo design). Origem: fatores_urbanos.csv (17 tipos canônicos prefixo INF-*).

EC EVENTO_CRIMINAL

Ato criminal denunciado ou registrado. Lógica law enforcement. Origem: disk_denuncia.csv (assuntos.*) + ocorrências. ~40 tipos canônicos (VIO-*, PAT-*, DRG-*, ARM-*, etc.).

VS VULNERABILIDADE_SOCIAL

Condição de vulnerabilidade de pessoas. Origem: FU tipo 19 (PSR) + CPSR + DD classes específicas. Tipos canônicos prefixo VSO-*.

Os 7 pontos de sobreposição entre as duas taxonomias (FU vs DD) representam onde condição ambiental + evento criminal existem no mesmo espaço — exatamente os hotspots de maior valor analítico pro CompStat:

Decisões a fechar antes de implementar:
  • Granularidade do grid: H3 res 9 (~150m hex) ou grid quadrado 250m? H3 é padrão pra urbano + lib pronta (h3-py), grid quadrado é mais simples.
  • Período: mensal (YYYY-MM) é seguro dado que data/hora estão vazios no dataset; semanal só se viabilizar.
  • Tipos de ocorrência: hoje 3 colunas (transeunte/celular/coletivo). Se aparecerem mais subtipos no desc_delito, adicionar coluna ou normalizar.
  • Score BINGO: fórmula candidata = norm(n_total_ocorrencias) × has_dinamica_relevante × norm(n_fatores_relevantes) com pesos. Validar com mentor.
Por que essa estrutura: SQL/pandas simples, fácil pro mentor + Prefeitura entenderem, time series por célula sai grátis, heatmap visualiza direto. Qualitativo (LLM) vive em tabelas separadas joinadas pela PK — não polui a fact table.

4. Detecção de inconsistências — regras + julgamento LLM

Duas camadas:

Regras programáticas (rápidas, determinísticas)

  • Geo: lat/long fora do bbox município RJ
  • Geo: coordenadas suspeitas de inversão (compare range)
  • Completude: campo obrigatório ausente / vazio
  • Encoding: caracteres inválidos em texto
  • Cross-source: bairro do registro não existe na lista canônica
  • Cross-source: câmera em área que não está no shapefile (Lauro Müller, Bangu)
  • Volume: zero ocorrências em área conhecidamente ativa
  • Temporal: data fora do período esperado
  • Duplicidade: mesma denúncia, geo diferente

Julgamento LLM (slow, contextual)

  • Conteúdo do RELINT consistente com padrões que extraímos de DD?
  • Denúncia diz que ocorre no bairro X mas relato menciona bairro Y
  • Padrão extraído de DD é generalizado o suficiente pra área? (overfitting de poucas amostras)
  • Confidence da extração abaixo do threshold
  • Recomendação contradiz exemplo do RELINT

6. Auditoria — como humano inspeciona

  • Cada registro consolidado vem com confidence por camada + source_log (de onde veio cada peça)
  • Decisões automáticas geram entrada em audit.jsonl: {timestamp, rule, before, after, confidence}
  • Decisões humanas vão pra mesma log com actor=human + reason
  • Pipeline reproduzível: rodar com mesmo input + mesmas decisões = mesmo output
  • UI mínima de revisão: listar pendências por severidade · ação inline (aceitar sugestão IA · corrigir manual · marcar como aceito-com-flag)

7. Implicações pro time

  • Não começar a calcular score antes da base consolidada estar pronta — score em base inconsistente é veneno
  • Logs de auditoria são parte da entrega — Prefeitura precisa saber por que cada conclusão foi tirada
  • Threshold de confiança parametrizável — não hardcode 0.6, deixe operador ajustar
  • LLM extraction de DD é a parte mais imprevisível — começar por aí pra calibrar prompts e ter buffer de tempo
  • "Auditável por humano" não significa "humano revisa tudo" — significa que pode revisar quando quiser
