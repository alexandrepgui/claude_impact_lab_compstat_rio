TL;DR — estratégia

Princípio mestre: "Feijão com arroz primeiro." O BINGO bem feito e confiável é o objetivo. Tudo o que vem depois — detecção de inconsistências, integração com Twitter, novas fontes — só ganha tempo de quem garantiu primeiro o core.

Core BINGO confiável + dinâmico

Cruzamento das 3 camadas (mancha · dinâmica · fator urbano relevante) feito de forma bem feita, reproduzível e atualizável ao longo do tempo. Áreas se movem — não tratamos como mapas estáticos.

Plus 1 Inconsistências + revisão humana

Forma parametrizada de detectar inconsistências nos dados e solicitar revisão humana quando confiança baixa. Não trate ruído como verdade.

Plus 2 Integração fácil de novas fontes

Arquitetura plugável pra absorver fontes adicionais sem reescrever. Exemplo concreto: Twitter / X (alinhado com Desafio Extra 1 do briefing oficial).

Princípio: feijão com arroz primeiro

"Antes de fazermos os plus a gente tem que fazer bem o feijão com arroz. O objetivo principal é o BINGO. O BINGO é o cruzamento dos dados de forma bem feita e confiável; a intenção das áreas não é estática, é dinâmica e se move ao longo do tempo."

Critério de decisão pra qualquer feature: melhora o BINGO core ou é decoração? Decoração entra só depois que o core estiver sólido (testado, com saída em .doc pra pelo menos 1 área completa). Risco de inversão: investir em Twitter/inconsistências antes do core = demo bonito sem substância = nota baixa em Impacto Real.

Visão ETL — onde está o trabalho real

Enquadrando como pipeline ETL, a alocação de esforço fica clara: Extract não é o problema (os dados já estão entregues). O peso fica no Transform — é onde mora o cruzamento das fontes e a garantia de qualidade. Load é importante mas direto (gerar .doc). Plus features se encaixam naturalmente em E ou L depois.

E Extract não é o gargalo

Os 4 CSVs + 1 shapefile + 8 RELINTs já estão no repo oficial. Ler bem (com os encodings/separadores certos do data inventory) é mecânica, não problema.

Plus futuro (E): adapters pra Twitter / 1746 BigQuery / outras fontes públicas. Só entra depois do core.

FOCO

T Transform

O problema mora aqui. Não é "limpar coluna" — é cruzar mancha × dinâmica × fator urbano de forma bem feita e confiável. Inclui:

  • Pareamento geoespacial (intersecções com polígonos FM)
  • Síntese qualitativa dos relatos (DD + RELINTs → padrões)
  • Filtragem do fator urbano pela dinâmica
  • Score do BINGO e recompute dinâmico quando shapefile muda
  • Garantia de qualidade entre fontes distintas (mais difícil que dentro de uma fonte)

L Load crítico

Mecânica de gerar .docx é resolvida (python-docx), mas o output É o produto — é o artefato que vai pra mesa da reunião de terça e que os jurados abrem em 3 min.

O que importa no Load:

  • Fidelidade ao template do anexo do briefing PDF (estrutura que a Duda já usa)
  • 4 norteadoras respondidas claramente no resumo executivo
  • Plano de ação acionável por órgão — não relatório, instrução
  • Mapa de calor + temporal + dinâmica embutidos no .doc (figuras, não links)
  • Honestidade analítica — onde confiança é baixa, sinaliza ao invés de esconder

Por que crítico: Impacto Real (40 pts) é avaliado pelo que vai pra prefeitura usar. Transform sem Load bom = análise correta que ninguém consome.

Plus futuro (L): dashboard interativo · endpoint API · entrega via email. Tudo depois do .doc sólido.

Ordem de batalha: Transform e Load andam em paralelo — ter BINGO confiável que produz .doc útil pra 1 área é o objetivo conjunto do feijão com arroz. Sem Load bom, Transform vira análise órfã. Sem Transform sólido, Load vira template vazio. Extract expansion (novas fontes) só depois disso.

Core — BINGO confiável e dinâmico

O que constitui "feito bem":

Camada 1

Mancha criminal — sólida

Heatmap espacial das 115k ocorrências agregado por mês/ano (granularidade real do dataset). Intersecção com polígonos das áreas FM via geometria. Não confiar em código AISP — pareamento espacial obrigatório.

Camada 2

Dinâmica criminal — qualitativa

Extração de padrões dos relatos: disk_denuncia.csv (texto livre em relato_redacted) + RELINTs (.docx). Sintetizar: modus operandi, rotas fuga, pontos recepção, horários, controle territorial. Por área.

Camada 3

Fator urbano — filtrado pela dinâmica

2.085 fatores já vêm com órgão responsável. Filtrar pelos relevantes: iluminação só importa se a dinâmica é noturna; calçadão só importa se é roubo a pé. Camada 3 só conta quando coincide com camada 2.

Output

Relatório Analítico em .doc

Por área. Estrutura definida pelo briefing: resumo executivo (4 norteadoras) + análise temporal + dinâmica + fatores relevantes + painel coincidências (BINGO) + plano de ação por órgão. 1 área completa > 8 áreas pela metade.

Dinâmica

Áreas não são estáticas

O briefing afirma: quando a FM aciona uma área, o crime migra. BINGO precisa ser recomputável a cada novo ciclo de dados. Sem hardcode dos 9/22 polígonos. Pipeline = entrada de dados → recálculo → novo relatório.

Critérios de "core pronto"

  • ✅ Pelo menos 1 área tem relatório .doc end-to-end gerado pelo pipeline (sem retrabalho manual)
  • ✅ As 3 camadas se conectam por geometria (não código)
  • ✅ Pipeline é reproduzível: rodar com dados novos = sai relatório novo
  • ✅ Saída faz sentido pro humano que valida (mentor Ramon ou Duda em revisão)
  • ✅ Sem hardcode de quais são as áreas — feed do shapefile ou parametrizado

Áreas dinâmicas — como tratar

O briefing é explícito: "polígonos não são estáticos... a gente também visa analisar esses dados para eventualmente atualizar esses novos polígonos, porque a gente sabe que o crime pode migrar uma vez que a gente está acionando um componente novo."

Onde isso aparece no pipeline

  • Polígonos como input parametrizado, não constante hardcoded
  • Pipeline reaceita shapefile atualizado sem código novo
  • Score de BINGO recalcula automaticamente quando shapefile muda

Possíveis abordagens (a discutir)

  • Detectar migração: ocorrências fora do polígono atual mas adjacentes
  • Sugerir novos polígonos baseado em cluster espacial recente
  • Alerta quando densidade fora-polígono > densidade dentro-polígono

Quem decide isso → time. Não escopear sem alinhamento.

Pipeline Transform — design detalhado

Princípio: pipeline 100% automatizada mas auditável por humanos. Cada passo deixa rastro. Inconsistências viram fila de revisão (auto-resolvida por IA quando seguro · escalada pra humano quando ambígua). Resultado: base consolidada confiável sobre a qual o score do BINGO é calculado.

1. Fluxo da pipeline (visão de alto nível)

P1

Load + Normalize (programático)

Ler cada fonte com seus encodings/separadores corretos. Padronizar CRS (WGS84), corrigir eixos invertidos (fatores_urbanos), renomear colunas pro schema canônico.

P2

Detect inconsistências (rule engine + LLM)

Rodar bateria de regras (geo bbox, completude, cross-source). LLM judga casos ambíguos. Output: lista de inconsistências com severidade + sugestão de correção.

P3

Resolve (auto-fix OU fila humana)

Se confiança alta + correção segura → auto-fix com log. Se ambíguo ou destrutivo → fila pra humano decidir. Pipeline pausa pra essa área até resolução.

P4

Enrich (programático + LLM)

Geo-intersect tudo com os polígonos FM. LLM extrai padrões de DD relatos + RELINTs (modus operandi, rotas fuga, recepção, horários, controle territorial). Filtra fatores urbanos pelos que importam pra dinâmica.

P5

Consolidate (programático)

Construir "área-snapshot" — registro único por (área FM × período) com todas as camadas integradas. Esse é o input pro score do BINGO.

P6

Audit trail (sempre)

Cada passo escreve: inconsistências detectadas, decisões tomadas (auto vs humano), confidence scores, timestamps. Humano pode reabrir e re-rodar.

4. Detecção de inconsistências — regras + julgamento LLM

Duas camadas:

Regras programáticas (rápidas, determinísticas)

  • Geo: lat/long fora do bbox município RJ
  • Geo: coordenadas suspeitas de inversão (compare range)
  • Completude: campo obrigatório ausente / vazio
  • Encoding: caracteres inválidos em texto
  • Cross-source: bairro do registro não existe na lista canônica
  • Cross-source: câmera em área que não está no shapefile (Lauro Müller, Bangu)
  • Volume: zero ocorrências em área conhecidamente ativa
  • Temporal: data fora do período esperado
  • Duplicidade: mesma denúncia, geo diferente

Julgamento LLM (slow, contextual)

  • Conteúdo do RELINT consistente com padrões que extraímos de DD?
  • Denúncia diz que ocorre no bairro X mas relato menciona bairro Y
  • Padrão extraído de DD é generalizado o suficiente pra área? (overfitting de poucas amostras)
  • Confidence da extração abaixo do threshold
  • Recomendação contradiz exemplo do RELINT

6. Auditoria — como humano inspeciona

  • Cada registro consolidado vem com confidence por camada + source_log (de onde veio cada peça)
  • Decisões automáticas geram entrada em audit.jsonl: {timestamp, rule, before, after, confidence}
  • Decisões humanas vão pra mesma log com actor=human + reason
  • Pipeline reproduzível: rodar com mesmo input + mesmas decisões = mesmo output
  • UI mínima de revisão: listar pendências por severidade · ação inline (aceitar sugestão IA · corrigir manual · marcar como aceito-com-flag)

7. Implicações pro time

  • Não começar a calcular score antes da base consolidada estar pronta — score em base inconsistente é veneno
  • Logs de auditoria são parte da entrega — Prefeitura precisa saber por que cada conclusão foi tirada
  • Threshold de confiança parametrizável — não hardcode 0.6, deixe operador ajustar
  • LLM extraction de DD é a parte mais imprevisível — começar por aí pra calibrar prompts e ter buffer de tempo
  • "Auditável por humano" não significa "humano revisa tudo" — significa que pode revisar quando quiser

Plus 1 Detecção de inconsistências + revisão humana

Update: esta capability deixou de ser "plus opcional" e virou parte do core Transform pipeline (passos P2 + P3). Mantida nesta seção como referência mas o conteúdo principal mora em Pipeline Transform — design detalhado.

Problema: dados têm ruído real (ex.: eixos invertidos em fatores_urbanos.csv, datas vazias em df_ocorrencias_tratado, encoding latin-1 em DD, áreas que aparecem em câmeras mas não no shapefile). Tratar tudo como verdade = recomendação errada.

Forma desejada (parametrizada)

  • Regras de validação configuráveis (range geográfico, completude de colunas, consistência cross-source)
  • Score de confiança por registro / por área
  • Quando confiança < threshold → fila de revisão humana (não esconder, expor)
  • Operador da prefeitura vê: "X registros precisam revisão antes de virar ação" — decide manter/descartar/corrigir

Inconsistências já identificadas (data inventory)

  • Áreas extras nas câmeras (Lauro Müller 50, Bangu Calçadão 30) sem RELINT ou shapefile
  • Eixos invertidos em fatores_urbanos.csv
  • 1 denúncia → várias linhas em DD (uma por órgão difundido)
  • Domínio territorial cobre estado inteiro, contamina análise se não filtrar bbox
Cuidado: Plus 1 não substitui o core. Inconsistência sem ação não vale. Pense "humano vê problema → resolve em 30s → pipeline continua". Não construir validação que ninguém vai usar.

Plus 2 Integração fácil de novas fontes (Twitter/X)

Premissa arquitetural: tratar fontes como adapters plugáveis. Cada fonte → normalizada para o formato interno → entra no pipeline de BINGO sem mudar lógica central.

Twitter como exemplo concreto

  • Alinhado com Desafio Extra 1 do briefing oficial (Inteligência de Redes Sociais)
  • Twitter/X é interessante porque: timing real, geolocalização aproximada (bairros), narrativa qualitativa que complementa DD/RELINTs
  • Adapter ideal: termos relevantes (FM, nomes de áreas, padrões de roubo) → georreferenciar → enriquecer dinâmica criminal

Outras fontes possíveis (futuras)

  • Central 1746 (BigQuery, já referenciada como camada de validação)
  • Dados climáticos / sensores meteorológicos (correlação clima × crime)
  • Mobilidade urbana (Mobilidade Rio, Uber, BRT) — fluxos de pedestre
  • Eventos (calendário Maracanã, COR Eventos) — picos sazonais
Não escopar Twitter no day 1 sem core pronto. Twitter API tem fricção (rate limit, auth), facilmente consome 2-3h. Só entra depois que pelo menos 1 área tem relatório completo end-to-end.

O que não fazemos (até o core estar pronto)

  • ❌ Adapter Twitter / scraping de redes sociais
  • ❌ Dashboard interativo bonito (output oficial é .doc)
  • ❌ ML model treinado do zero — usar Claude pra síntese qualitativa, não pra classificação supervisionada
  • ❌ Análise de domínio territorial faccional como feature (alto risco político, baixo valor pro BINGO)
  • ❌ Cobertura das 22 áreas (só 8 têm dados — focar nelas)
  • ❌ Censo PSR como camada principal (é só 1 dos 20 fatores, não trate como protagonista)
